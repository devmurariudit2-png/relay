## 2024-05-17 - [Optimize Reconciliation O(N²) loop to O(N) map lookups]
**Learning:** Found an O(N²) nested loop bottleneck in the `reconcileEngine.js` when fuzzy-matching transactions by amounts and dates. Using `Math.round(amount * 100)` to create integer keys for grouping float-based prices into a map gives you O(1) lookups on the target group. However, because floating point variation and rounding differences can cause matches to span across an integer boundary (e.g. 100.001 vs 100.006 rounding to 10000 vs 10001 despite a diff of 0.005 being < 0.01 threshold), it is absolutely critical to look up boundaries (`key - 1`, `key`, `key + 1`) to guarantee correct logic while maintaining O(N) speed.
**Action:** When converting float-based matching O(N²) loops into O(N) hash map lookups grouped by rounding, always check immediate adjacent keys to avoid missing valid fuzzy matches that cross rounding boundaries.
